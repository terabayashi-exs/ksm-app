# 大会GO！ UIデザイン基本方針

> **本文書の目的**: Claude Codeで開発を行うスタッフが、UIに関する判断で迷わず、統一感のあるインターフェースを構築するための基本方針。  
> **対象**: 開発チーム全員（Claude Codeのプロジェクト指示としても使用）  
> **最終更新**: 2026年2月

---

## 1. デザイン原則（迷ったときの判断基準）

以下の5原則は優先順位順。矛盾する場合は上位の原則を優先する。

### 原則1: モバイルファースト・現場ファースト
- すべてのUIはスマートフォン（幅375px）から設計を開始する
- PC表示はモバイルの拡張であり、モバイルでの体験を劣化させてPC版を作らない
- 炎天下・片手操作・手袋着用でも使えることを前提とする

### 原則2: 即座にわかる・即座に使える
- 初見のユーザーが説明なしに操作できるUIを目指す
- 主要操作は3タップ以内で完了する
- 操作結果は必ず視覚的フィードバックを即座に返す（ローディング、成功、エラー）

### 原則3: 情報の階層が明確である
- 画面ごとに「一番伝えたいこと」が1つだけ明確にある
- 視線の流れ（上→下、左→右）に沿って情報の優先度が下がる
- 装飾で情報の優先度を曖昧にしない

### 原則4: 一貫性を保つ
- 同じ意味のUIは、どの画面でも同じ見た目・同じ挙動にする
- 新しいパターンを作る前に、既存のパターンで対応できないか確認する
- コンポーネント名・クラス名・変数名の命名規則を統一する

### 原則5: スポーツの活気を表現する
- デザインは「堅すぎず、軽すぎず」を基調とする
- 信頼感（協会・組織が使うシステム）と親しみやすさ（子供の保護者も使う）の両立
- スポーツの躍動感をカラーとタイポグラフィで表現する

## 2. デザイントークン

### 2.1 カラーパレット

Tailwind CSSのカスタムカラーとして定義する。ハードコードされたカラーコードは使用禁止。

```
【プライマリ】大会GO!のブランドカラー
primary-50:   #f0f9ff   ← 背景のうっすらとした色付け
primary-100:  #e0f2fe   ← ホバー時の背景
primary-500:  #0ea5e9   ← メインのブランドカラー（ボタン、リンク、アクセント）
primary-600:  #0284c7   ← ホバー時のボタン
primary-700:  #0369a1   ← アクティブ時のボタン
primary-900:  #0c4a6e   ← テキストとしてのブランドカラー

【セカンダリ】補助カラー（大会のカテゴリ分けなどに使用）
secondary-500: #8b5cf6  ← 紫系アクセント

【セマンティック（意味のある色）】※ 必ずこの用途にのみ使う
success:   #22c55e   ← 成功・完了・勝利
warning:   #f59e0b   ← 注意・進行中
error:     #ef4444   ← エラー・失敗・敗北
info:      #3b82f6   ← 情報・お知らせ

【ニュートラル（グレースケール）】
gray-50:   #f9fafb   ← ページ背景
gray-100:  #f3f4f6   ← カード背景、セクション区切り
gray-200:  #e5e7eb   ← ボーダー、divider
gray-300:  #d1d5db   ← disabled状態のボーダー
gray-400:  #9ca3af   ← プレースホルダーテキスト
gray-500:  #6b7280   ← 補足テキスト、キャプション
gray-600:  #4b5563   ← セカンダリテキスト
gray-800:  #1f2937   ← プライマリテキスト（本文）
gray-900:  #111827   ← 見出しテキスト
```

#### カラー使用ルール

| 用途 | 使うべき色 | 禁止 |
|------|-----------|------|
| CTAボタン（主要アクション） | primary-500 → hover: primary-600 | 赤系、グレー系 |
| 破壊的操作（削除、取消） | error（赤） | primary（青） |
| テキストリンク | primary-500 | 下線なしのリンク |
| 成功トースト | success背景 + 白テキスト | primary背景 |
| 背景色 | gray-50（ページ）/ white（カード） | 色付きの背景は最小限に |
| 試合結果：勝利 | success | - |
| 試合結果：敗北 | error | - |
| 試合結果：引分 | gray-500 | - |

> **注**: 上記のカラーコードは暫定値。ブランドカラーの最終決定後に置き換える。  
> Tailwindのconfigで `colors` を一元定義し、コード内で直接hex値を書かない。

### 2.2 タイポグラフィ

```
【フォントファミリー】
表示用（見出し・ロゴ周辺）: "M PLUS Rounded 1c", "Zen Maru Gothic", sans-serif
本文用: "Noto Sans JP", "Hiragino Sans", sans-serif
数値・スコア表示: "Inter", "Noto Sans JP", sans-serif（等幅感のある数字が重要）

【フォントサイズスケール】（Tailwindクラス名で統一）
text-xs:   0.75rem (12px)  ← キャプション、注釈
text-sm:   0.875rem (14px) ← 補足テキスト、テーブル内
text-base: 1rem (16px)     ← 本文（これが基準）
text-lg:   1.125rem (18px) ← 小見出し、強調テキスト
text-xl:   1.25rem (20px)  ← セクション見出し
text-2xl:  1.5rem (24px)   ← ページ見出し（モバイル）
text-3xl:  1.875rem (30px) ← ページ見出し（PC）
text-4xl:  2.25rem (36px)  ← ヒーローセクション見出し

【行間】
本文: leading-relaxed (1.625)
見出し: leading-tight (1.25)
UIラベル: leading-normal (1.5)

【太さ】
normal (400): 本文
medium (500): UIラベル、ナビゲーション
semibold (600): 小見出し、強調
bold (700): ページ見出し、重要な数値（スコアなど）
```

#### タイポグラフィ使用ルール
- 1画面内で使用するフォントサイズは最大4段階まで
- 太字（bold）は見出しとスコア表示にのみ使い、本文中の強調には `semibold` を使う
- 全角英数字は使用しない（半角に統一）
- スコア表示（得点など）は常に `font-mono` または `tabular-nums` を適用する

### 2.3 スペーシング

```
【基本単位】4px（Tailwindのデフォルト: 1 = 0.25rem = 4px）

【よく使う余白】
p-1 (4px):   アイコン周り、バッジ内部
p-2 (8px):   タグ、チップ内部
p-3 (12px):  コンパクトなカード内部
p-4 (16px):  標準的なカード内部、セクション内部（モバイル）
p-6 (24px):  カード内部（PC）、セクション間
p-8 (32px):  ページ上下の余白

【セクション間の余白】
モバイル: space-y-4 (16px) 〜 space-y-6 (24px)
PC: space-y-6 (24px) 〜 space-y-8 (32px)

【カード間の余白】
gap-3 (12px): モバイル
gap-4 (16px): PC
```

#### スペーシングルール
- 余白は必ずTailwindのスペーシングスケールを使う（`p-4`, `mt-6`など）
- `margin-top: 13px` のような中途半端な値は禁止
- 要素間の余白は `gap` または `space-y/x` で管理し、個別マージンは最小限にする
- コンテンツエリアの最大幅: `max-w-4xl`（896px）を標準とする

### 2.4 角丸・シャドウ

```
【角丸】
rounded-md (6px):   ボタン、入力フィールド、バッジ
rounded-lg (8px):   カード、モーダル、ドロップダウン
rounded-xl (12px):  大きなカード（特集・ヒーロー要素）
rounded-full:       アバター、丸いアイコンボタン

【シャドウ】
shadow-sm:   カード（デフォルト状態）
shadow-md:   カード（ホバー時）、ドロップダウンメニュー
shadow-lg:   モーダル、フローティング要素
shadow-none: フラットなUI要素（テーブル行など）
```

## 3. コンポーネントルール

すべてshadcn/uiをベースとし、カスタムコンポーネントは必要最小限とする。

### 3.1 ボタン

```
【バリエーションと使い分け】

default（primary）: メインのアクション（保存、登録、送信）
  → bg-primary text-white
  → 1画面に1つだけ（最も重要なアクション）

secondary: 補助的なアクション（キャンセル、戻る、フィルター適用）
  → bg-secondary text-secondary-foreground

destructive: 破壊的操作（削除、退会）
  → bg-destructive text-white
  → 必ず確認ダイアログを挟む

outline: 優先度の低いアクション（詳細を見る、もっと読む）
  → border border-input bg-background

ghost: ナビゲーション内・テーブル内のアクション
  → 背景なし、ホバーで薄い背景

link: テキストリンクとして振る舞うボタン
  → 下線付きテキスト
```

#### ボタンサイズルール
```
sm:   テーブル内、コンパクトなUI（高さ32px）
default: 標準的なフォーム（高さ40px）
lg:   主要CTA、モバイルでの主操作（高さ48px、最小タップ領域44px確保）
```

#### ボタン共通ルール
- ボタンラベルは動詞始まり（「保存する」「削除する」「大会を作成」）
- アイコン付きボタンは、アイコン（左）+ テキスト の配置を標準とする
- ローディング中はボタンを `disabled` にし、スピナーを表示する
- モバイルでのタップ領域は最低44x44pxを確保する

### 3.2 フォーム

```
【入力フィールド共通】
- ラベルは入力欄の上に配置（左揃え）
- 必須項目は ラベル + 赤い「*」で表示
- プレースホルダーはヒント（例: "チーム名を入力"）として使い、ラベルの代替にしない
- エラーメッセージは入力欄の直下に赤字で表示する
- 入力中のフィールドは border-primary で囲む

【バリデーション】
- Zod スキーマで一元定義し、フロント・サーバー両方で検証する
- リアルタイムバリデーションは blur 時（フィールドを離れた時）に発火
- 送信時に全フィールドを再検証する
- エラーメッセージは具体的に（「入力してください」ではなく「チーム名は必須です」）

【フォームレイアウト】
- モバイル: 1カラム（full width）
- PC: 最大2カラム（関連するフィールドのみ横並び、例: 姓+名）
- 送信ボタンはフォームの末尾、右揃え（モバイルはfull width）
```

### 3.3 カード

```
【標準カード】
- bg-white rounded-lg shadow-sm border border-gray-200
- padding: p-4（モバイル）/ p-6（PC）
- ホバー時: shadow-md + translateY(-1px) でわずかに浮く演出（リンクカードのみ）

【大会カード】（大会一覧で使用）
- 上部にステータスバッジ（募集中/開催中/完了）
- 大会名は text-lg font-semibold で最も目立つ
- 主催者・日程・会場は text-sm text-gray-600
- 部門数はバッジで表示

【試合結果カード】
- チーム名は左右対称に配置
- スコアは text-2xl font-bold font-mono で中央に大きく
- 勝利チーム側に success カラーのアクセント
```

### 3.4 テーブル

```
【基本ルール】
- ヘッダー行: bg-gray-50 font-medium text-gray-600 text-sm
- データ行: hover:bg-gray-50 で行全体がハイライト
- ボーダー: 行間に border-b border-gray-200（縦罫線は使わない）
- セル内のテキスト: text-sm
- 数値は右揃え、テキストは左揃え

【モバイル対応】
- 横スクロール禁止（モバイルでテーブルが横にはみ出してはいけない）
- モバイルではテーブルをカード形式に変換するか、重要列のみ表示する
- 順位表（リーグ）は横スクロール許可（例外）だが、チーム名列は固定する
```

### 3.5 モーダル・ダイアログ

```
【使う場面】
- 確認が必要な破壊的操作（削除、取消）
- 簡単な入力フォーム（項目3つ以下）
- 重要な通知（エラー、完了）

【使わない場面】
- 複雑なフォーム → 別ページに遷移する
- 情報の表示のみ → トーストまたはページ内で表示
- ネストされたモーダル（モーダルの中からモーダルを開かない）

【デザインルール】
- オーバーレイ: bg-black/50（半透明黒）
- モーダル本体: bg-white rounded-lg shadow-lg max-w-md
- 閉じる手段: ×ボタン + オーバーレイクリック + Escキー（すべて対応）
- アクションボタン: 右下配置、キャンセル（左）+ 確定（右）
```

### 3.6 トースト・通知

```
【用途別】
success: 操作完了（保存しました、登録しました）→ 3秒で自動消去
error:   操作失敗（保存に失敗しました）→ 手動で閉じるまで残る
warning: 注意喚起（締切が近づいています）→ 5秒で自動消去
info:    お知らせ（新しい大会が追加されました）→ 5秒で自動消去

【位置】
モバイル: 画面上部中央
PC: 画面右上

【ルール】
- 同時表示は最大3つまで（古いものから消える）
- トーストにアクションボタンを含めない（ただし「取り消す」のUndoは例外的に可）
```

### 3.7 ナビゲーション

```
【ヘッダー】
- 固定ヘッダー（sticky top-0）
- 高さ: h-16（64px）
- 内容: ロゴ（左）+ ナビリンク（中央/右）+ ユーザーメニュー（右端）
- モバイル: ロゴ（左）+ ハンバーガーメニュー（右）

【モバイルメニュー】
- ハンバーガーメニューからスライドイン（右から）
- オーバーレイ付き
- メニュー項目は十分な高さ（48px以上）でタップしやすく

【タブナビゲーション】
- 大会詳細ページなど、同一コンテキスト内の切り替えに使用
- アクティブタブは border-bottom + primary color で明示
- スクロール可能タブ: タブが多い場合は横スクロール（インジケーター付き）

【パンくずリスト】
- 管理画面でのみ使用（一般向けページでは不要）
- トップ > 大会管理 > 大会名 > 試合管理 のような階層表示
```

## 4. レイアウトパターン

### 4.1 ページ構成の基本

```
【一般向け（観覧者・チーム代表者）】
┌──────────────────────────────┐
│ ヘッダー（固定）               │
├──────────────────────────────┤
│                              │
│  コンテンツエリア              │
│  max-w-4xl mx-auto px-4      │
│                              │
├──────────────────────────────┤
│ フッター                      │
└──────────────────────────────┘

【管理画面（大会管理者・サイト管理者）】
┌──────────────────────────────┐
│ ヘッダー（固定）               │
├──────┬───────────────────────┤
│      │                       │
│ サイド│  コンテンツエリア       │
│ バー  │  max-w-5xl             │
│      │                       │
│      │                       │
├──────┴───────────────────────┤
│ （フッターなし or 最小限）     │
└──────────────────────────────┘

```

### 4.2 レスポンシブブレークポイント

```
sm:  640px   ← 大きめのスマートフォン
md:  768px   ← タブレット（ここでサイドバー表示切替）
lg:  1024px  ← 小型PC
xl:  1280px  ← 標準PC

原則: sm/md/lg の3段階で対応する。xl以上は lg と同じレイアウト。
```

### 4.3 グリッドシステム

```
【カード一覧】
モバイル(default): grid-cols-1
sm:               grid-cols-2（大会カードなど）
lg:               grid-cols-3

【ダッシュボード統計カード】
モバイル(default): grid-cols-2
lg:               grid-cols-4

【試合結果一覧】
すべてのサイズ: grid-cols-1（試合カードは常に1列）
```

## 5. 画面タイプ別ガイドライン

### 5.1 一般公開ページ（大会一覧、大会詳細、試合結果）

- **目的**: 情報をすばやく見つけて理解する
- **トーン**: 明るく、スポーティ、見やすい
- **背景**: gray-50
- **カードベース**: 情報はカードで区切り、スキャンしやすく
- **CTA**: 「チーム登録」「大会に申し込む」を最も目立たせる

### 5.2 管理画面（大会管理者向け）

- **目的**: 効率的にデータを入力・管理する
- **トーン**: 機能的、プロフェッショナル、落ち着いた
- **背景**: white（サイドバーはgray-50）
- **テーブル多用**: データ管理はテーブル形式を基本とする
- **バルク操作**: 複数選択+一括操作をサポートする

### 5.3 試合進行画面（QRコード経由の審判入力）

- **目的**: 最速でスコアを入力する
- **トーン**: 高コントラスト、大きなUI、ミニマル
- **特別ルール**:
  - フォントサイズを通常の1.5倍にする
  - ボタンは最低56x56pxのタップ領域
  - 不要な要素は徹底的に排除する
  - 入力ミスを防ぐ確認ステップを必ず挟む
  - 屋外での視認性のため、高コントラスト配色

### 5.4 トップページ（ランディング）

- **目的**: サービスを理解し、適切な入口に導く
- **3つの入口を明確に**: 管理者ログイン / チーム代表者ログイン / チーム登録
- **大会情報をファーストビューに**: 開催中・募集中の大会数を目立たせる
- **特徴紹介はスクロール後**: まず「使う」ためのUIを優先

## 6. アニメーション・トランジション

```
【基本方針】
- アニメーションは「意味のある変化」にのみ使う
- 装飾的なアニメーションは原則使わない
- 所要時間は 150ms 〜 300ms の範囲内

【使うべきアニメーション】
- ページ遷移: フェードイン (opacity 0→1, 150ms)
- モーダル表示: フェードイン + スケール (scale 0.95→1, 200ms)
- トースト表示: スライドイン (translateY, 200ms)
- ホバー効果: カードのわずかな浮き (translateY -1px, 150ms)
- アコーディオン開閉: 高さのトランジション (200ms)

【使わないアニメーション】
- バウンス、揺れ、回転などの過度な演出
- 3秒以上かかるアニメーション
- 自動再生のスライドショー（ユーザー操作で切り替え）
- スクロールに連動するパララックス
```

## 7. アイコン

```
【ライブラリ】
lucide-react を標準とする（shadcn/uiとの親和性）

【サイズ】
16px (w-4 h-4): テキスト内、バッジ内
20px (w-5 h-5): ボタン内、ナビゲーション内
24px (w-6 h-6): 単独アイコン、カード内
32px (w-8 h-8): 空状態のイラスト的アイコン

【ルール】
- アイコンだけのボタンには必ず aria-label を設定する
- アイコン + テキスト の場合、アイコンはテキストの左側に配置
- 色はテキストと同じ色を使う（別の色でアイコンを装飾しない）
```

## 8. ダークモード対応方針

```
【現時点の方針】
- 既存のダークモード（テーマ切り替え機能）は維持する
- ただし、ダークモードの改善・最適化は優先度を下げる
  → コア機能の開発・ライトモードのUI統一を優先する
- 新規画面・コンポーネント作成時のルール:
  - 色はすべてCSS変数（Tailwindのテーマ）で管理する
  - ハードコードされた色（bg-white, text-blackなど）は使わず、
    セマンティックなクラス（bg-background, text-foreground）を使う
  - shadcn/uiのテーマ機能をそのまま活用する
  - これにより、ダークモードが「壊れない」状態を最低限保つ
- ダークモード専用のデザイン調整（色味の微調整、コントラスト最適化など）は
  コア機能が安定した段階で改めて取り組む
```

## 9. アクセシビリティ最低基準

```
【必須対応】
- カラーコントラスト比: WCAG AA準拠（テキスト 4.5:1以上、大文字 3:1以上）
- すべてのインタラクティブ要素にフォーカスインジケーターを表示
- 画像には alt テキストを設定（装飾画像は alt="" で空にする）
- フォーム要素には label を紐付ける
- キーボードだけで主要操作が完了できる
- スクリーンリーダーで情報が正しく読み上げられる

【推奨対応】
- aria-live でスコア更新を通知
- 色だけでなくアイコン・テキストでも状態を伝える（色覚多様性対応）
```

## 10. 命名規則

```
【コンポーネントファイル名】
PascalCase: TournamentCard.tsx, MatchResultCard.tsx

【コンポーネント配置】
共通UI: components/ui/（shadcn/uiベース）
機能固有: components/tournament/, components/match/, components/team/
レイアウト: components/layout/

【CSSクラス（Tailwind）】
- ユーティリティクラスの順序: レイアウト → サイズ → 余白 → 装飾 → テキスト
  例: "flex items-center gap-2 p-4 bg-white rounded-lg text-sm text-gray-600"
- 長くなる場合は cn() ユーティリティで結合
- カスタムCSSクラスは原則作らない（Tailwindで表現する）

【変数名・Props】
camelCase: tournamentName, matchResult, teamCount
booleanは is/has/can 接頭辞: isActive, hasScore, canEdit
```

## 11. 品質チェックリスト

新しい画面・コンポーネントを作成する際に確認する。

### 全画面共通
- [ ] モバイル（375px幅）で表示崩れがないか
- [ ] タブレット（768px幅）で適切なレイアウトになっているか
- [ ] タップ領域が44x44px以上確保されているか
- [ ] ローディング状態が表示されるか
- [ ] エラー状態が適切に表示されるか
- [ ] 空状態（データなし）の表示があるか
- [ ] カラーはデザイントークンのみ使用しているか
- [ ] フォントサイズはTailwindのスケールのみ使用しているか

### フォーム
- [ ] バリデーションエラーが入力欄の直下に表示されるか
- [ ] 必須項目に赤い「*」があるか
- [ ] 送信中はボタンがdisabledになり、スピナーが表示されるか
- [ ] 送信成功後にトーストまたはリダイレクトがあるか

### テーブル・一覧
- [ ] モバイルで横スクロールが発生していないか（例外: 順位表）
- [ ] 空状態の表示があるか
- [ ] ソート・フィルターが機能しているか（該当する場合）

## 12. Claude Codeへの指示テンプレート

上記の方針をClaude Codeのプロジェクト指示に組み込む際の要約版：

```
【UIデザイン指示（Claude Code用）】

■ 技術スタック
- Next.js 15 (App Router) + React 19 + TypeScript
- Tailwind CSS 4 + shadcn/ui
- アイコン: lucide-react
- フォーム: React Hook Form + Zod

■ 基本原則
1. モバイルファースト（375px幅から設計）
2. タップ領域は最低44x44px
3. 色・フォントサイズ・余白はすべてTailwindのテーマトークンを使用
4. ハードコードされたカラーコード禁止
5. 新しいUIパターンを作る前に既存パターンで対応できるか確認

■ コンポーネント
- UIコンポーネントはshadcn/uiを使う
- カスタムCSSは原則禁止（Tailwindユーティリティで解決）
- ボタンのバリエーション: default(CTA), secondary, destructive, outline, ghost
- 破壊的操作には必ず確認ダイアログ

■ レイアウト
- コンテンツ幅: max-w-4xl mx-auto px-4
- 管理画面はサイドバー付きレイアウト
- グリッド: モバイル1列、sm2列、lg3列

■ フォーム
- ラベルは入力欄の上、エラーは入力欄の下
- バリデーションはblur時に発火
- 送信中はボタンをdisabledにしスピナー表示

■ 禁止事項
- 横スクロールが発生するテーブル（順位表を除く）
- モーダルの中からモーダルを開く
- 3秒以上のアニメーション
- アイコンだけのボタン（aria-labelなし）
- bg-white, text-black等のハードコード色（bg-background, text-foregroundを使う）
```
